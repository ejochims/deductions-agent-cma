01 · The problem

The money is already gone

When a retailer short-pays an invoice, the difference is a deduction — usually claimed against a trade promotion. The cash left with the remittance; what remains is the work of deciding, claim by claim, whether it should stay gone.

Every claim is an investigation

Validating one deduction means reconstructing what was promised (the contract), what ran (the promo calendar), and what actually sold (POS scans) — then checking history for duplicates. That costs an analyst minutes to hours, per claim.

The economics leak

Small claims get written off because checking them costs more than the claim is worth. Large ones get settled under deadline pressure. Both paths quietly hand margin to the retailer.

The bar for automation is high

A wrong approval pays an invalid claim; a wrong denial strains a retailer relationship. An agent doing this work has to be verifiably careful — which is exactly why it makes a strong proving ground for agent engineering.

The case set is synthetic — a fictional manufacturer, Meridian Foods, closing Q1 FY26 — but shaped like a real desk: clean approvals, aggressive re-bills, a duplicate, inflated scan counts, missing paperwork.

02 · The approach

The tool boundary is the security boundary

The agent reasons on Anthropic's orchestration layer, but its seven tools — six reads and one write gate — are fulfilled host-side — the host answers each custom_tool_use from fixtures/, and nothing is mounted into the sandbox (no egress, no filesystem). Even memory is just another host-fulfilled tool.

04 · Platform

Why Claude Managed Agents

The agent loop runs on Anthropic's servers. This repo ships no orchestration machinery — just a thin event stream (src/run_agent.py), the tool fulfilment layer, and the eval harness wrapped around them. Three platform properties carry the design:

Hosted loop

Sessions on demand

One session per case and trial. The session goes idle at every tool call and waits for the host to answer — and that pause is where the draft-only gate lives.

Immutable versions

Evals that can't go stale

Every config update publishes a new agent version, and the runner fingerprints agent.yaml — each score traces to the exact prompt that produced it. Editing the prompt auto-publishes and re-pins before the next run.

Declarative sandbox

An environment that starts empty

The environment config mounts nothing and allows no egress. Every capability arrives as a custom tool the host fulfils, so fixtures stay local and the answer key stays out of reach.

One honest friction: the platform's native memory store assumes file tools that this deliberately empty sandbox lacks. That mismatch caused a silent 0% you'll see two slides ahead — and the fix reused the same custom-tool pattern as everything else.

05 · The eval harness

The harness is validated before any result is believed

Five graders

Deterministic checks per draft: action correct, amount within tolerance, evidence cited, no hallucinated evidence, and a safety hard-fail — threshold respected (never settle > $10k).

Two calibration gates

Gate A: all 18 reference solutions score perfect. Gate B: a blind "approve-everything" agent fails every judgement bucket. When known-good scores perfect and known-bad lands at the floor, the ruler measures what it claims.

A calibrated judge

The LLM judge must fail three planted bad justifications before it is allowed to grade. It records rationale quality but never overrides the programmatic pass/fail.

pass^k reads harder than an average: a case counts as passing only if all three trials got it right — the metric that matters when the failure mode is a confident wrong settlement.

06 · Refinement

The eval caught a silent failure — then measured the fix

The memory bucket scored 0% pass^3 while every other bucket looked healthy. Nothing threw: the session status stayed ok and the drafts looked plausible. A native memory store was mounted at /mnt/memory/, but the deliberately toolless sandbox exposed no way to read it — every access returned "Unknown tool" and the agent drafted the precedent cases blind. A happy-path demo sails straight past this; the per-bucket, ground-truth eval made it impossible to miss.

The fix: retire the native store and serve recall the same way as every other capability — a host-fulfilled get_precedents tool. Re-running the eval measured the climb:

07 · Refinement

Remove a piece, measure what breaks

Each experiment removes one part of the system — a prompt section or a tool — re-runs the eval, then reverts. If the score drops, that part was doing real work, and the grader that catches the drop is named. If it doesn't, that's an honest negative worth keeping.

10 · What generalizes

Three design decisions worth stealing

The tool boundary is the security boundary

Because nothing is mounted, the answer key can't leak into context and untrusted remittance text can't reach the filesystem. Access control is a property of the architecture itself.

The tool layer enforces policy

The $10k threshold and the null-amount coercion on a deny are enforced in the tool layer. The model can be wrong or adversarially steered and the money still can't move — the worst case is a bad draft a human reviews.

Validation is a first-class artifact

Graders, a calibrated judge, calibration gates, and a regression harness ship alongside the agent — the difference between a plausible demo and a defensible one, and what lets a finance owner sign off on auto-settling.

11 · Beyond the POC

What we'd revisit at production scale

Deliberately simplified to prove the trust layer. Three decisions we'd reopen under production constraints, each recorded with the condition that would reopen it.

Memory model

Curated recall → governed learning

Today precedents are a host-fulfilled, read-only tool — no model-writable state, no poisoning surface. If the agent should accumulate new precedents over time, move to native memory with human-gated writes, backed by write→recall and poisoning cases in the eval.

Systems of record

Fixtures → live integrations

The six read tools read synthetic fixtures. Production swaps their implementations for retailer EDI / deduction portals, the ERP (e.g. SAP), and OCR of backup docs — behind the same typed tool interface. The boundary holds; only the data source changes.

Governance & audit

Draft-only → approval workflow

The $10k human gate is the seed. At scale: role-based execution, an immutable audit trail, and adversarial remittance text as a first-class eval bucket, so prompt injection shows up in the score with a number attached.

Each is captured as an architecture decision in docs/decisions/, with a concrete “when to revisit” trigger.
